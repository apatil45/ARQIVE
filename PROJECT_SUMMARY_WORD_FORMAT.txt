ARQIVE PROJECT - QUICK INTERVIEW REFERENCE
==========================================

ELEVATOR PITCH (30 seconds)
----------------------------

ARQIVE (Audit-document Retrieval and Query Intelligence Via Embeddings) is a 
fully local RAG (Retrieval-Augmented Generation) system that enables users to 
upload documents and query them using AI. It runs entirely on a user's machine 
without requiring cloud services, ensuring privacy and data security.


================================================================================
ARCHITECTURE OVERVIEW
================================================================================

Frontend (Next.js/React/TypeScript)
    ↓ HTTP/REST
Backend (FastAPI/Python)
    ↓
    ├── ChromaDB (Vector Storage)
    ├── SQLite (Metadata)
    ├── Ollama (LLM)
    └── Sentence-Transformers (Embeddings)

KEY TECHNOLOGIES:
-----------------
• Backend: FastAPI, Python 3.9+, async/await
• Frontend: Next.js 14, React 18, TypeScript
• Vector DB: ChromaDB with HNSW indexing
• Metadata DB: SQLite with WAL mode
• Embeddings: BAAI/bge-small-en-v1.5 (384 dimensions)
• LLM: Ollama (tinyllama for CPU, configurable)


================================================================================
CORE WORKFLOWS
================================================================================

DOCUMENT INGESTION FLOW
------------------------
1. User uploads file (PDF/DOCX/TXT)
2. File validation (size, type, security)
3. Save to storage (S3 or local filesystem)
4. Extract text (PyPDF2, python-docx)
5. Chunk text (400 tokens, 50 token overlap)
6. Generate embeddings (sentence-transformers)
7. Store in ChromaDB (vector DB)
8. Store metadata in SQLite
9. Return document ID

QUERY FLOW (RAG)
----------------
1. User submits query
2. Sanitize query (prevent prompt injection)
3. Check query result cache
4. Generate query embedding
5. Vector search in ChromaDB (cosine similarity)
6. Filter by user access permissions
7. Apply similarity threshold (0.6)
8. Select diverse, high-quality chunks
9. Build context (max 1500 tokens for CPU)
10. Generate answer with Ollama LLM
11. Return answer with citations
12. Cache result for future queries


================================================================================
SECURITY FEATURES
================================================================================

1. Authentication: JWT tokens (30 min expiry)
2. Authorization: Role-based access control (Admin, Auditor, Viewer)
3. Password Security: bcrypt hashing
4. File Security: Magic byte validation, filename sanitization, path traversal 
   prevention
5. Input Validation: Prompt injection detection, query sanitization
6. Rate Limiting: 100 queries/hour per IP
7. Audit Logging: All security events logged
8. CORS: Configured for specific origins


================================================================================
PERFORMANCE OPTIMIZATIONS
================================================================================

CACHING
-------
• Query embedding cache (LRU, 1000 entries)
• Query result cache (1 hour TTL)
• User document access cache (5 minutes TTL)

CPU OPTIMIZATION
----------------
• Reduced context tokens (1500 vs 4000)
• Adaptive max_results (1-3 chunks based on query complexity)
• Early stopping for context building
• Chunk truncation for long documents
• Hardware-aware timeouts (30s GPU, 60s CPU)

DATABASE OPTIMIZATION
---------------------
• SQLite WAL mode for concurrency
• Indexes on frequently queried columns
• Optimized queries (JOINs, only fetch needed data)
• Connection pooling ready

VECTOR SEARCH
-------------
• HNSW indexing in ChromaDB
• Similarity threshold filtering (0.6)
• Diversity-based chunk selection


================================================================================
DATABASE SCHEMA
================================================================================

USERS TABLE
-----------
• username (PRIMARY KEY)
• password_hash
• role (admin, auditor, viewer)
• email, full_name
• created_at

DOCUMENTS TABLE
---------------
• id (PRIMARY KEY, UUID)
• filename
• file_path (S3 key or local path)
• file_type
• uploaded_by (FOREIGN KEY → users)
• uploaded_at
• metadata (JSON: chunk_count, file_size, storage_type)

DOCUMENT ACCESS TABLE
----------------------
• document_id (FOREIGN KEY → documents)
• username (FOREIGN KEY → users)
• access_level (read, write)
• PRIMARY KEY (document_id, username)

================================================================================
KEY DESIGN DECISIONS
================================================================================

WHY FASTAPI?
------------
• Built-in async support
• Automatic API documentation
• Type hints and validation
• Better performance than Flask

WHY CHROMADB?
-------------
• Local-first (no cloud needed)
• HNSW indexing for fast search
• Easy Python integration
• Good performance for local use

WHY SQLITE?
-----------
• Simple setup, no server needed
• Sufficient for local use
• Easy to migrate to PostgreSQL later

WHY OLLAMA?
-----------
• Runs locally, no API keys
• Privacy-focused
• Multiple model support
• Easy to configure

WHY NEXT.JS?
------------
• Server-side rendering
• Built-in routing
• Better performance
• TypeScript support


================================================================================
CHALLENGES SOLVED
================================================================================

SLOW QUERY PERFORMANCE
-----------------------
Problem: CPU inference was very slow
Solution: Reduced context size, adaptive chunking, caching, hardware detection

OLLAMA TIMEOUTS
---------------
Problem: Long queries timed out
Solution: Hardware-aware timeouts, context limits, better error handling

MEMORY ISSUES
-------------
Problem: Large documents consumed too much memory
Solution: Streaming, chunking, lazy loading, early stopping

ACCESS CONTROL
---------------
Problem: Users could access unauthorized documents
Solution: Document-level filtering, role-based access, metadata filtering


================================================================================
SCALABILITY CONSIDERATIONS
================================================================================

CURRENT LIMITATIONS
-------------------
• Single SQLite instance
• Local ChromaDB
• Single Ollama instance
• No load balancing

HOW TO SCALE
------------
1. Database: Migrate to PostgreSQL, add read replicas
2. Vector DB: Use managed service (Pinecone) or distributed ChromaDB
3. Backend: Horizontal scaling with load balancer
4. Caching: Redis for distributed caching
5. LLM: Use faster models, GPU acceleration, or cloud APIs
6. Storage: S3 for document storage
7. Queue: Celery/RQ for async document processing
