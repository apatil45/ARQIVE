1. SYSTEM DESIGN & ARCHITECTURE
================================================================================

BASIC QUESTIONS
---------------

Q1. What is ARQIVE and what problem does it solve?
    Expected Answer: ARQIVE is a local RAG (Retrieval-Augmented Generation) 
    system for document intelligence. It allows users to upload documents and 
    query them using AI without cloud dependencies.

Q2. Walk me through the high-level architecture of ARQIVE.
    Expected Answer: Frontend (Next.js) → Backend (FastAPI) → Vector DB 
    (ChromaDB) + SQLite + Ollama LLM. Documents are chunked, embedded, and 
    stored. Queries are embedded, searched, and answered using RAG.

Q3. Why did you choose a microservices-like architecture with separate frontend 
    and backend?
    Expected Answer: Separation of concerns, independent scaling, technology 
    flexibility, easier testing.

Q4. What are the main components of your system and how do they interact?
    Expected Answer: Frontend (UI), Backend API, Vector Database (ChromaDB), 
    SQLite (metadata), Embedding Model (sentence-transformers), LLM (Ollama).

INTERMEDIATE QUESTIONS
----------------------

Q5. How would you scale this system to handle 10,000 concurrent users?
    Expected Answer: Load balancers, horizontal scaling of FastAPI instances, 
    Redis for caching, database connection pooling, CDN for static assets, 
    async processing for document ingestion.

Q6. What happens when a user uploads a document? Describe the flow.
    Expected Answer: File validation → Save to storage (S3/local) → Extract 
    text → Chunk text → Generate embeddings → Store in ChromaDB → Store 
    metadata in SQLite → Return document ID.

Q7. How would you handle system failures? What's your disaster recovery plan?
    Expected Answer: Database backups, replication, health checks, graceful 
    degradation, retry mechanisms, monitoring and alerting.

Q8. Why did you choose ChromaDB over other vector databases?
    Expected Answer: Local-first, easy setup, HNSW indexing for fast similarity 
    search, good Python integration, no external dependencies.

ADVANCED QUESTIONS
------------------

Q9. How would you design this system to be distributed across multiple servers?
    Expected Answer: Stateless backend instances, shared vector DB (or 
    distributed), database replication, message queue for async tasks, service 
    discovery.

Q10. What are the bottlenecks in your current architecture?
     Expected Answer: Single-threaded Ollama inference, sequential document 
     processing, no caching layer initially, SQLite for production scale.

Q11. How would you implement real-time collaboration features (multiple users 
     querying simultaneously)?
     Expected Answer: WebSockets for real-time updates, optimistic UI updates, 
     conflict resolution, event sourcing.

2. BACKEND DEVELOPMENT (PYTHON/FASTAPI)
================================================================================

BASIC QUESTIONS
---------------

Q12. Why did you choose FastAPI over Flask or Django?
     Expected Answer: Built-in async support, automatic API documentation, type 
     hints, better performance, modern Python features.

Q13. Explain how async/await works in your FastAPI endpoints.
     Expected Answer: Async functions allow non-blocking I/O operations, 
     enabling concurrent request handling. Used for database queries, file 
     operations, and LLM calls.

Q14. What is dependency injection in FastAPI? Show an example from your code.
     Expected Answer: get_current_user dependency for authentication. FastAPI 
     automatically resolves dependencies and injects them into route handlers.

Q15. How do you handle CORS in your application?
     Expected Answer: Using CORSMiddleware with configurable origins, 
     credentials, methods, and headers.

INTERMEDIATE QUESTIONS
----------------------

Q16. Explain your error handling strategy. How do you handle exceptions 
     globally?
     Expected Answer: Global exception handler, validation error handler, 
     proper HTTP status codes, logging, user-friendly error messages in 
     production.

Q17. What is rate limiting and why did you implement it?
     Expected Answer: Prevents abuse, DoS protection. Using slowapi with 100 
     queries/hour per IP. Protects expensive LLM operations.

Q18. How does your streaming endpoint work? Why is it useful?
     Expected Answer: Server-Sent Events (SSE) for real-time token streaming 
     from Ollama. Better UX, perceived performance, handles long-running 
     queries.

Q19. Explain the lifespan context manager in FastAPI.
     Expected Answer: Handles startup (DB initialization) and shutdown logic. 
     Ensures proper resource cleanup.

Q20. How do you validate file uploads? What security measures are in place?
     Expected Answer: File size limits, magic byte validation, filename 
     sanitization, path traversal prevention, type checking.

ADVANCED QUESTIONS
------------------

Q21. How would you implement request queuing for document ingestion?
     Expected Answer: Celery/RQ for async task processing, Redis as message 
     broker, worker processes, progress tracking.

Q22. Explain how you would implement API versioning.
     Expected Answer: URL versioning (/api/v1/), header versioning, or separate 
     routers. Maintain backward compatibility.

Q23. How do you handle database migrations in your system?
     Expected Answer: Alembic for SQLAlchemy, or manual migration scripts. 
     Version control, rollback capability.

Q24. What is the difference between async def and def in FastAPI? When would 
     you use each?
     Expected Answer: async def for I/O-bound operations (DB, API calls). def 
     for CPU-bound or synchronous operations.

4. DATABASE & DATA STORAGE
================================================================================

BASIC QUESTIONS
---------------

Q39. Why did you use both SQLite and ChromaDB? What's the purpose of each?
     Expected Answer: SQLite for structured metadata (users, documents), 
     ChromaDB for vector embeddings and similarity search. Different use cases.

Q40. What is a vector database? How does it differ from a relational database?
     Expected Answer: Stores high-dimensional vectors, optimized for similarity 
     search. Relational DB stores structured data with relationships.

Q41. Explain the schema of your SQLite database.
     Expected Answer: Users table (username, password_hash, role), Documents 
     table (id, filename, file_path, metadata), Document_access table 
     (document_id, username, access_level).

Q42. What are indexes and why did you create them?
     Expected Answer: Speed up queries. Created indexes on uploaded_by, 
     uploaded_at, document_id, username for faster lookups.

Q43. What is WAL mode in SQLite? Why is it beneficial?
     Expected Answer: Write-Ahead Logging. Better concurrency, allows readers 
     and writers simultaneously, faster writes.

INTERMEDIATE QUESTIONS
----------------------

Q44. How would you migrate from SQLite to PostgreSQL for production?
     Expected Answer: Schema migration, connection string changes, data 
     migration script, testing, rollback plan.

Q45. Explain your document access control implementation.
     Expected Answer: Document_access table for sharing, filtering by 
     uploaded_by or explicit access, admin override, efficient queries with 
     JOINs.

Q46. How do you handle database connection pooling?
     Expected Answer: aiosqlite handles connections, but for production would 
     use connection pool, limit connections, proper cleanup.

Q47. What is the difference between SQLite and PostgreSQL? When would you choose 
     each?
     Expected Answer: SQLite is file-based, single-user, embedded. PostgreSQL 
     is server-based, multi-user, ACID-compliant, better for production.

Q48. How would you implement full-text search in SQLite?
     Expected Answer: FTS5 extension, virtual tables, ranking, relevance 
     scoring, combined with vector search.

ADVANCED QUESTIONS
------------------

Q49. How would you implement database replication and failover?
     Expected Answer: Master-slave replication, read replicas, automatic 
     failover, data consistency, monitoring.

Q50. Explain how you would shard your vector database.
     Expected Answer: Partition by document_id hash, user_id, or time. 
     Distributed ChromaDB or use managed service like Pinecone.

Q51. What is eventual consistency? How does it apply to your system?
     Expected Answer: Vector DB and SQLite may be out of sync temporarily. 
     Eventual consistency acceptable for some operations.

Q52. How would you implement database transactions for document ingestion?
     Expected Answer: Begin transaction, insert to SQLite, insert to ChromaDB, 
     commit both or rollback on failure.

5. RAG & AI/ML CONCEPTS
================================================================================

BASIC QUESTIONS
---------------

Q53. What is RAG? Explain how it works in your system.
     Expected Answer: Retrieval-Augmented Generation. Query → Embed → Vector 
     Search → Retrieve Chunks → Build Context → Generate Answer with LLM.

Q54. What are embeddings? How are they generated in your system?
     Expected Answer: Dense vector representations of text. Using 
     sentence-transformers (BAAI/bge-small-en-v1.5), 384 dimensions, L2 
     normalized.

Q55. What is cosine similarity? Why is it used for vector search?
     Expected Answer: Measures angle between vectors, not magnitude. Good for 
     semantic similarity, normalized vectors, range -1 to 1.

Q56. Explain the document chunking strategy. Why chunk documents?
     Expected Answer: LLMs have context limits. Chunk size 400 tokens, overlap 
     50 tokens. Preserves context, enables precise retrieval.

Q57. What is the difference between semantic search and keyword search?
     Expected Answer: Semantic uses meaning/embeddings, keyword uses exact 
     matches. Semantic finds related concepts, keyword finds exact terms.

INTERMEDIATE QUESTIONS
----------------------

Q58. How do you select the best chunks from search results?
     Expected Answer: Similarity threshold filtering (0.6), diversity (max 1 
     chunk per document), highest similarity first, quality checks.

Q59. What is prompt engineering? How did you optimize your prompts?
     Expected Answer: Crafting effective prompts for LLM. Different prompts 
     for simple fact queries vs complex analytical queries, clear instructions.

Q60. Explain the similarity threshold. How did you choose 0.6?
     Expected Answer: Filters low-relevance chunks. 0.6 balances precision and 
     recall. Higher = stricter, lower = more results.

Q61. How would you improve RAG accuracy?
     Expected Answer: Better chunking (semantic, hierarchical), re-ranking, 
     hybrid search (BM25 + vector), query expansion, fine-tuning embeddings.

Q62. What is the difference between embedding models? Why BAAI/bge-small-en-v1.5?
     Expected Answer: Size vs quality tradeoff. BGE-small is fast, 384-dim, 
     good for local inference. Larger models = better but slower.

Q63. How do you handle context window limits?
     Expected Answer: Token counting, early stopping, truncation, prioritizing 
     relevant chunks, max 1500 tokens for CPU optimization.

ADVANCED QUESTIONS
------------------

Q64. How would you implement hybrid search (keyword + semantic)?
     Expected Answer: BM25 for keyword matching, vector search for semantic, 
     combine scores with weighted fusion, re-rank results.

Q65. What is re-ranking? How would you implement it?
     Expected Answer: Cross-encoder model to re-rank top-k results. More 
     accurate but slower. Use after initial retrieval.

Q66. How would you handle multi-modal documents (images, tables)?
     Expected Answer: Vision models for images, OCR, table extraction, 
     multi-modal embeddings, separate processing pipelines.

Q67. Explain query expansion and query rewriting techniques.
     Expected Answer: Synonym expansion, query decomposition, LLM-based 
     rewriting, user feedback loop, A/B testing.

Q68. How would you implement fine-tuning for domain-specific embeddings?
     Expected Answer: Collect domain data, fine-tune sentence-transformers, 
     evaluate on test set, A/B test, gradual rollout.

6. SECURITY & AUTHENTICATION
================================================================================

BASIC QUESTIONS
---------------

Q69. How does JWT authentication work in your system?
     Expected Answer: Login → verify credentials → generate JWT with 
     username/role → client stores token → send in Authorization header → 
     verify on each request.

Q70. What information is stored in your JWT tokens?
     Expected Answer: Username (sub), role, expiration time (exp). Signed with 
     secret key using HS256 algorithm.

Q71. Why is password hashing important? What algorithm do you use?
     Expected Answer: Prevents plaintext storage. Using bcrypt via passlib. 
     Salted, slow hashing, resistant to rainbow tables.

Q72. What is role-based access control (RBAC)? How is it implemented?
     Expected Answer: Admin, Auditor, Viewer roles. Decorators check roles, 
     database stores role, token includes role but verified from DB.

Q73. How do you prevent SQL injection attacks?
     Expected Answer: Parameterized queries with aiosqlite, no string 
     concatenation, input validation, ORM-like patterns.

INTERMEDIATE QUESTIONS
----------------------

Q74. What security vulnerabilities exist in your current implementation?
     Expected Answer: Default admin credentials, token in memory (could use 
     httpOnly cookies), no rate limiting on login, no 2FA, no token refresh.

Q75. How would you implement token refresh?
     Expected Answer: Short-lived access tokens, long-lived refresh tokens, 
     separate endpoint, secure storage, rotation on use.

Q76. Explain prompt injection attacks. How do you prevent them?
     Expected Answer: User input injected into LLM prompts. Sanitization, 
     pattern detection, input validation, prompt templates, output filtering.

Q77. What is CORS and why is it a security concern?
     Expected Answer: Cross-Origin Resource Sharing. Prevents unauthorized 
     domains from accessing API. Configured origins, credentials handling.

Q78. How do you handle file upload security?
     Expected Answer: File size limits, type validation (magic bytes), filename 
     sanitization, path traversal prevention, virus scanning (future).

Q79. What is the principle of least privilege? How does it apply to your roles?
     Expected Answer: Users get minimum permissions needed. Viewer = read-only, 
     Auditor = upload+query, Admin = full access.

ADVANCED QUESTIONS
------------------

Q80. How would you implement OAuth2/SSO?
     Expected Answer: OAuth2 flow, identity provider (Google, GitHub), token 
     exchange, user mapping, session management.

Q81. What is rate limiting and how would you implement it more robustly?
     Expected Answer: Per-user limits, sliding window, distributed rate limiting 
     (Redis), different limits per endpoint, whitelisting.

Q82. How would you implement audit logging for compliance?
     Expected Answer: Already implemented! Log all security events, immutable 
     logs, retention policies, compliance reporting, SIEM integration.

Q83. Explain how you would implement encryption at rest and in transit.
     Expected Answer: TLS/HTTPS for transit, database encryption, file 
     encryption, key management (AWS KMS), certificate management.

Q84. How would you handle session hijacking?
     Expected Answer: Secure token storage, short expiration, token rotation, 
     IP validation, device fingerprinting, anomaly detection.

7. PERFORMANCE & OPTIMIZATION
================================================================================

BASIC QUESTIONS
---------------

Q85. What caching strategies did you implement?
     Expected Answer: Query embedding cache (LRU, 1000 entries), query result 
     cache (1 hour TTL), user document access cache (5 min TTL).

Q86. Why is async programming important for performance?
     Expected Answer: Non-blocking I/O, concurrent request handling, better 
     resource utilization, scales better than threads.

Q87. What optimizations did you make for CPU inference?
     Expected Answer: Reduced context tokens (1500), adaptive max_results, 
     early stopping, chunk truncation, smaller model (tinyllama).

Q88. How does HNSW indexing improve vector search performance?
     Expected Answer: Hierarchical Navigable Small World. Approximate nearest 
     neighbor, logarithmic search time, much faster than linear scan.

Q89. What is the difference between eager and lazy loading?
     Expected Answer: Eager = load immediately, Lazy = load on demand. Embedding 
     model uses lazy loading to reduce startup time.

INTERMEDIATE QUESTIONS
----------------------

Q90. How would you optimize database queries?
     Expected Answer: Indexes, query optimization, connection pooling, read 
     replicas, query caching, batch operations.

Q91. Explain your adaptive max_results strategy.
     Expected Answer: Simple queries → fewer chunks (1-2), complex queries → 
     more chunks (2-3). Reduces processing time for simple queries.

Q92. What is the N+1 query problem? How would you prevent it?
     Expected Answer: Multiple queries in a loop. Use JOINs, batch loading, 
     eager loading, data fetching strategies.

Q93. How would you implement database connection pooling?
     Expected Answer: Pool of reusable connections, limit max connections, 
     timeout handling, health checks, proper cleanup.

Q94. What is memoization? Where did you use it?
     Expected Answer: Cache function results. Used in React (useMemo, 
     useCallback), Python (embedding cache, query result cache).

ADVANCED QUESTIONS
------------------

Q95. How would you implement a distributed cache (Redis)?
     Expected Answer: Redis cluster, key expiration, cache invalidation 
     strategies, cache warming, monitoring, fallback to database.

Q96. What is database sharding? How would you implement it?
     Expected Answer: Partition data across multiple databases. Shard by 
     user_id, document_id hash, or time. Consistent hashing, routing logic.

Q97. How would you optimize for GPU inference?
     Expected Answer: Batch processing, model quantization, TensorRT 
     optimization, mixed precision, pipeline parallelism, model serving.

Q98. Explain your early stopping strategy for context building.
     Expected Answer: Stop adding chunks when target token count reached (500 
     tokens), minimum threshold (50 tokens), prevents exceeding limits.

Q99. How would you implement request batching?
     Expected Answer: Collect multiple requests, batch process, return results. 
     Reduces overhead, improves throughput.

Q100. What is the difference between horizontal and vertical scaling?
      Expected Answer: Horizontal = more servers, Vertical = more powerful 
      server. Horizontal is preferred for web apps, better cost and reliability.
