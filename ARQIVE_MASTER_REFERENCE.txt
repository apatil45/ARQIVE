SECTION 1 — PROJECT IDENTITY & ELEVATOR PITCH
--------------------------------------------------------------------------------

NAME: ARQIVE
FULL NAME: Audit-document Retrieval and Query Intelligence Via Embeddings

ONE-LINER:
ARQIVE is a fully local RAG (Retrieval-Augmented Generation) system that lets
users upload documents and query them using AI—everything runs on the user's
machine with no cloud required, ensuring privacy and data security.

30-SECOND PITCH:
ARQIVE (Audit-document Retrieval and Query Intelligence Via Embeddings) is a
fully local RAG system that enables users to upload documents (PDF, DOCX, TXT)
and query them using AI. It runs entirely on a user's machine without requiring
cloud services or external APIs. It combines vector search (ChromaDB),
sentence-transformers embeddings, and a local LLM (Ollama) with role-based
access control and audit-aware document handling—targeted at audit, compliance,
and financial document analysis.

TAGLINE:
"Smart document assistant that finds answers in your documents using AI—private,
local, and secure."


SECTION 2 — BUSINESS CONTEXT, INITIATIVE & NOVELTY
--------------------------------------------------------------------------------

BUSINESS CONTEXT:
- Document intelligence for audit, compliance, and financial teams.
- Target users: auditors, executives, compliance officers, and viewers who need
  to query internal documents (audit reports, financial statements, management
  letters) without sending data to the cloud.
- Use cases: Q&A over audit documents, executive summaries, finding findings/
  recommendations, financial metrics extraction, opinion and KAM (Key Audit
  Matters) lookup, compliance checks.

INITIATIVE:
- Built as a full-stack, local-first document Q&A platform to demonstrate
  end-to-end RAG, embeddings, vector search, and LLM integration.
- Evolved with production-oriented improvements: configuration centralization,
  structured logging, connection pooling, cache invalidation, JWT verification,
  two-phase commit for ingestion, query history, document preview, and
  industry-standard audit enhancements (audit chunking, metadata extraction,
  query classification, specialized prompts).

NOVELTY / DIFFERENTIATORS:
1. Fully local: no cloud APIs; embeddings (sentence-transformers) and LLM
   (Ollama) run on-device—suitable for sensitive/audit documents.
2. Audit-aware RAG: structure-preserving chunking (sections, hierarchy),
   audit metadata extraction (opinion, KAMs, findings, compliance, dates),
   query classification (11 types: financial metrics, opinion, findings,
   compliance, summary, KAM, comparison, risk, recommendations, metadata,
   general), and role-based query routing (executive/auditor/viewer).
3. Document-level access control: vector search results filtered by user
   permissions; RBAC (Admin, Auditor, Viewer) with document ownership and
   explicit access.
4. Production-minded design: SQLite connection pooling, three-level caching
   (embedding, query result, user-document), hardware-aware timeouts (CPU/GPU),
   prompt injection mitigation, rate limiting, audit logging, and optional S3
   storage.
5. End-to-end ownership: FastAPI backend, Next.js/React/TypeScript frontend,
   ChromaDB + SQLite, ingestion pipeline (extract, chunk, embed, store) with
   two-phase commit and rollback on failure.


SECTION 3 — TECHNICAL STACK & ARCHITECTURE
--------------------------------------------------------------------------------

BACKEND:
- Language: Python 3.9+
- Framework: FastAPI (async/await)
- Auth: JWT (python-jose, HS256, 30 min expiry), bcrypt password hashing
- Metadata DB: SQLite (aiosqlite), WAL mode, connection pool (5 base, 10 overflow)
- Vector DB: ChromaDB, HNSW index, cosine similarity
- Embeddings: sentence-transformers, BAAI/bge-small-en-v1.5, 384 dimensions, L2
  normalized
- LLM: Ollama (local), e.g. tinyllama for CPU; configurable model
- Document parsing: PyPDF2 (PDF), python-docx (DOCX), UTF-8/Latin-1 (TXT)
- Chunking: tiktoken (cl100k_base), 400 tokens, 50 overlap; optional
  audit-aware chunker (structure/section preservation)
- Storage: Local filesystem or optional S3 (boto3)
- Rate limiting: slowapi, 100 queries/hour per IP
- Logging: Structured logging, request IDs (X-Request-ID), optional JSON logs
- Audit: Dedicated audit logger (logins, uploads, deletions, queries,
  suspicious activity)

FRONTEND:
- Framework: Next.js 14, React 18
- Language: TypeScript
- Styling: CSS Modules
- HTTP: Axios; auth token in localStorage + in-memory cache; backend token
  verification (/auth/verify)
- Pages: /login, /chat (streaming SSE), /upload, /documents, /admin (role-
  restricted), index

ARCHITECTURE (HIGH LEVEL):
  Frontend (Next.js/React/TS) — HTTP/REST —> Backend (FastAPI)
    Backend uses: ChromaDB (vectors), SQLite (users, documents, access,
    query_history), Ollama (LLM), sentence-transformers (embeddings),
    local/S3 (file storage)


SECTION 4 — FEATURES & CAPABILITIES
--------------------------------------------------------------------------------

CORE FEATURES:
- User authentication (login, JWT, token verification)
- Role-based access: Admin (full access, user management), Auditor (upload +
  query own/shared docs), Viewer (query shared docs only)
- Document upload: PDF, DOCX, TXT; max 50MB; magic-byte validation; local or S3
- Document ingestion: extract text → chunk (token-based or audit-aware) →
  embed → store in ChromaDB + metadata in SQLite; two-phase commit with rollback
- RAG query: sanitize query → (optional) result cache → classify query → embed
  → vector search → filter by user access → similarity threshold → diversity
  selection → build context → generate answer via Ollama → return answer +
  citations; streaming (SSE) or non-streaming
- Document list (with pagination), document preview (first N chunks), document
  deletion
- Query history: save query, answer preview, document IDs, response time; GET
  and DELETE endpoints
- Admin: list users, create user (admin only)
- Health check: SQLite, ChromaDB, Ollama, disk/memory (optional psutil)

AUDIT-SPECIFIC:
- Audit-aware chunking: preserve sections, hierarchy, financial tables; detect
  audit sections (Executive Summary, Opinion, KAMs, Findings, etc.)
- Audit metadata extraction: document type, period, auditor, auditee, opinion
  type, KAMs, financial metrics, compliance standards, industry
- Query classification: 11 types with confidence and retrieval strategy
- Specialized prompts per query type and role (executive/auditor/viewer)
- Query-type-specific similarity thresholds and max_results

SECURITY:
- JWT + backend verification; bcrypt; RBAC; document-level access; prompt
  injection detection; query/file validation; rate limiting; audit logging;
  no sensitive detail in production error responses

SECTION 6 — CHALLENGES SOLVED & DESIGN DECISIONS
--------------------------------------------------------------------------------

CHALLENGES SOLVED:
1. Slow CPU inference: reduced context size, adaptive chunk count, caching,
   hardware-aware timeouts, early stopping, chunk truncation.
2. Ollama timeouts: configurable GPU/CPU timeouts, context limits, clearer
   errors.
3. Memory/large docs: streaming, chunking, lazy embedding load, early stop.
4. Unauthorized document access: document-level filtering in vector search,
   RBAC, metadata filters, cache invalidation on access changes.
5. RAG returning irrelevant text: stronger prompts (“understand question
   first”), metadata-query detection (titles, filenames), higher similarity
   threshold for non-metadata, better context labeling.
6. Token loss on refresh (external browser): token in localStorage + in-memory
   cache; backend verification.
7. Ingestion consistency: two-phase commit; rollback vector DB, SQLite, and
   files on failure.
8. Stale cache after access changes: cache invalidation on grant/revoke/delete.

KEY DESIGN DECISIONS:
- FastAPI: async, OpenAPI, validation, performance.
- ChromaDB: local, HNSW, simple integration.
- SQLite: no separate server; path to PostgreSQL later.
- Ollama: local LLM, no API keys, privacy.
- Next.js: SSR, routing, TypeScript.
- Audit-aware RAG: better accuracy and citations for audit/finance use cases.
